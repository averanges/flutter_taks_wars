# Flutter Task Wars

A mobile task management application developed according to a Figma design.

The project is built using **Clean Architecture**, with separation into `presentation / domain / data`, dependency injection via `get_it` + `injectable`, and **BLoC-based state management**.

## Architecture

### Project Structure

Main modules:

- `core/`

  - `environments/` – environment definitions
  - `usecase/` – base classes for use cases
  - `utils/` – utility classes

- `data/`

  - `datasources/`
    - `local/`
      - `auth_data_provider.dart` – works with `SharedPreferences` (saving / reading email)
    - `remote/`
      - `client/rest_client.dart` – Retrofit client built on top of `Dio`
      - `entities/request/` – request DTOs
      - `entities/response/` – response DTOs

  - `repository/` – repository implementations

- `domain/`

  - `entities/` – domain models isolated from API formats
  - `repository/` – abstractions for the data layer
  - `usecases/` – use cases

- `presentation/`

  - `blocs/` – feature-specific BLoCs
  - `navigation/`
    - `app_router.dart` – route declarations
    - `routes.dart` – wrapper around `GoRouter`
  - `pages/` – application screens
  - `widget/` – reusable UI components

- `resources/`

  - `themes/app_colors.dart` – color palette
  - `themes/app_text_style.dart` – text styles
  - `generated/fonts.gen.dart` – autogenerated fonts file (`FlutterGen`)

- `di/`

  - `injector.dart` – DI entry point (`getIt`, `configureDependencies`)
  - `modules/` – DI modules:
    - `logger_module.dart` – `Talker` + `TalkerDioLogger` for logging
    - `rest_client_module.dart` – configures `Dio` with baseUrl from `AppEnvironment` + logger integration
    - `shared_prefs_module.dart` – `SharedPreferences`

- `app.dart`

  - `AppWidget` – root `MaterialApp.router` with connected `GoRouter`

- `main.dart`

  - Flutter initialization, DI configuration, and launching `AppWidget`

### Data Flow

The simplified flow looks like this:

1. **UI (Widgets / Pages)**  
   Triggers BLoC events.

2. **BLoC (presentation layer)**

   - handles Events;
   - invokes corresponding UseCases from the `domain` layer;
   - emits States observed by UI widgets (`BlocBuilder`, `BlocListener`).

3. **UseCase (domain layer)**

   - encapsulates the business logic of a specific operation;
   - calls methods of abstract repositories (`TasksRepository`, `AuthRepository`).

4. **Repository (data layer)**

   - delegates work to `datasources`:
     - `RestClient` (remote);
     - `AuthDataProvider` / `SharedPreferences` (local).
   - maps DTOs (response / request) to domain models and vice versa (`TaskResponse ↔ TaskDomain`).

5. **DataSources**

   - remote: API calls through `Dio` + Retrofit;
   - local: key-value storage (email) in SharedPreferences.

6. **Returning to UI**

   - result flows as  
     `UseCase → BLoC → State → UI rendering (`BlocBuilder`) / side effects (`BlocListener` + dialogs)`.

### Error Handling and Validation

#### Error Handling

- All repositories wrap calls in `try/catch` and delegate exceptions to `ErrorHandler.handle(...)`.

- `ErrorHandler`:
  - If the error is already an `ApiException` — it is simply rethrown.
  - If it is a `DioException` — it attempts to parse the response body as `NetworkErrorResponse` (`error` field) → `NetworkError`.
  - If it is a `SocketException` / `HandshakeException` — returns `NetworkError(error: 'no_internet')`.
  - In all other cases — returns `NetworkError(error: 'unknown')`.

- `ApiException` contains:
  - `NetworkError networkError`
  - `StackTrace stackTrace`

- On the presentation layer, BLoCs (`TasksBloc`, `TaskBloc`) catch `ApiException` and convert it into `TasksErrorState` / `TaskErrorState` with a `networkError` field.

- In `HomePage` and `TaskPage`, errors are shown through `BlocListener` using dialogs via `DialogHelper.showErrorDialog(...)`, while users see a localized message from `networkError.userMessage`.

#### Validation

`FieldValidator` utility:

- `required(String? value, [String? fieldName])` – checks for empty value
- `email(String? value)` – basic email validation using RegExp
- `combine([...])` – sequentially runs a list of validators
- `validateEmail(String raw)` – convenient combined email validator (required + format)

Validation failure classes (all inherit from `ValidationFailure`):

- `RequiredFieldFailure` – "`<field> is required`"
- `EmptyEmailFailure`
- `InvalidEmailFormatFailure`
- `UnknownFailure`

## State Management

### Principles

The project uses **BLoC (Business Logic Component)** based on the `flutter_bloc` / `bloc` packages:

- **unidirectional data flow**:  
  `Event → BLoC → State → UI`

- **separation of business logic from UI**:
  - UI is declarative and “thin”
  - all logic for task list rendering, task creation, status updates, and sign-in lives inside BLoCs and UseCases

- **testability**:
  - BLoCs can be tested in isolation by mocking UseCases/repositories

- **scalability**:
  - each feature has its own BLoC, Events, and States
  - easy to add new screens and scenarios without affecting existing modules

This approach was chosen instead of simpler solutions (such as `setState`, `Provider`) because:

- the project has a clear layered architecture and already uses the UseCase/Repository pattern;
- structured async handling is needed (API requests, SharedPreferences storage);
- it allows centralized error handling (`ApiException` / `NetworkError`) and clean UI error presentation.

### BLoCs in the Project

1. **AuthBloc** (`presentation/blocs/auth/auth_bloc.dart`)

- Purpose: initializes authentication state at app startup.

- Events:
  - `AuthInitEvent` – triggered on app launch / opening `LogInPage`.

- States:
  - `AuthInitialState` → `AuthLoadingState` → `AuthLoadedState(email)` or `AuthErrorState`

- Logic:
  - calls `GetEmailUseCase`
  - if an email exists → navigates to the task list
  - if not → shows the login screen

---

2. **SignInBloc** (`presentation/blocs/sign_in/sign_in_bloc.dart`)

- Purpose: handles email input and persistence.

- Events:
  - `SignInSubmittedEvent(email)`

- States:
  - `SignInInitialState`
  - `SignInLoadingState`
  - `SignInSuccessState`
  - `SignInErrorState`

- Logic:
  - validates email via `FieldValidator.validateEmail`
  - calls `SignInWithEmailUseCase` to save email in `AuthRepository` (`SharedPreferences`)
  - on success:
    - emits `SignInSuccessState`
    - closes the dialog
    - navigates to `HomePage`

---

3. **TasksBloc** (`presentation/blocs/tasks/tasks_bloc.dart`)

- Purpose: manages the **task list** and filters.

- Events:
  - `TasksLoadEvent({TaskType? category, bool showLoading = true})`  
    loads / reloads tasks

  - `TasksChangeStatusEvent(task)`  
    changes task status

- States:
  - `TasksInitialState`
  - `TasksLoadingState`
  - `TasksLoadedState`
  - `TasksErrorState`  
    (contains tasks + category + optional `NetworkError`)

- Logic:
  - `GetTasksUseCase.execute()` – fetches task list
  - `ChangeTaskStatusUsecase.call(task)` – updates task status
  - in case of failure:
    - transitions to `TasksErrorState`
    - keeps previous task list so the UI is not cleared

---

4. **TaskBloc** (`presentation/blocs/task/task_bloc.dart`)

- Purpose: handles **creation / deletion** of individual tasks.

- Events:
  - `TaskSubmitEvent(task)` – create / update task
  - `TaskDeleteEvent(taskId)` – delete task

- States:
  - `TaskInitialState`
  - `TaskLoadingState`
  - `TaskSuccessState`
  - `TaskErrorState` (`NetworkError?`)

- Logic:
  - `CreateTaskUsecase(task)` – creates task
  - `DeleteTaskUsecase(taskId)` – deletes task
  - on error:
    - emits `TaskErrorState` with `NetworkError`
    - UI displays an error dialog

## Navigation

Navigation is built with **GoRouter**.

- Configuration: `presentation/navigation/routes.dart`

```dart
GoRouter(
  initialLocation: AppRouter.initial,
  routes: AppRouter.routes,
  errorBuilder: (context, state) => const NotFoundPage(),
);
```

- Routes are defined in `AppRouter`:

### `AppRouter.initial = '/initial'`
Login screen:

- creates `AuthBloc`

```dart
inject<AuthBloc>()..add(AuthInitEvent())
```

- displays `LogInPage`

### `AppRouter.tasks = '/tasks'`
Main task list screen:

- creates `TasksBloc`
- `HomePage` dispatches `TasksLoadEvent()` immediately in `initState`

### `AppRouter.creationTask = '/creation_task'`
Task creation / editing screen:

- creates `TaskBloc`
- `state.extra` passes `TaskDomain?` for editing

If `null`:

```dart
TaskDomain.empty(...)
```

is created for a new task.

### Navigation helpers

- `AppRouter.pop(context)`  
  Safe back navigation; if `canPop() == false`, redirects to `/tasks`.

## Environment and DI

### AppEnvironment

- `core/environments/app_environment.dart`  
  abstraction containing `tasksBaseUrl`

- `core/environments/development_environment.dart`

```dart
@LazySingleton(as: AppEnvironment)
@dev
class DevelopmentEnvironment implements AppEnvironment {
  @override
  String tasksBaseUrl = 'https://to-do.softwars.com.ua';
}
```

Using the `@dev` annotation and DI, additional environments (`prod`, `stage`, etc.) can be added by replacing `tasksBaseUrl`.

### Dependency Injection

- Uses `get_it` + `injectable`.

- `di/injector.dart`:
  - `configureDependencies(String environment)` – initializes all dependencies (generates `injector.config.dart`)
  - `inject<T>()` – shorthand wrapper over `getIt.get<T>()`

### DI Modules

- `LoggerModule` provides:

  - `Talker` – logger
  - `TalkerDioLogger` – integration with `Dio`

- `RestClientModule`:

  - creates `Dio` with:

```dart
BaseOptions(baseUrl: environment.tasksBaseUrl)
```

- registers `TalkerDioLogger` as an interceptor

- creates:

```dart
RestClient(Dio dio)
```

- `SharedPrefsModule`

  - uses `@preResolve` for initializing `SharedPreferences` before app startup

DI is initialized in `main.dart`:

```dart
Future<void> main() async {
  WidgetsFlutterBinding.ensureInitialized();
  GoRouter.optionURLReflectsImperativeAPIs = true;

  await configureDependencies(Environment.dev);

  runApp(const AppWidget());
}
```

## HTTP API and Local Storage

### HTTP API (Remote Data Source)

- Client: `RestClient` using Retrofit annotations  
(`@RestApi`, `@GET`, `@POST`, etc.)

- Base URL is taken from:

```dart
AppEnvironment.tasksBaseUrl
```

- Serialization / deserialization is handled via `json_serializable`
(generated `*.g.dart` files).

### Main methods

- `getTasks(email)`  
  → `TasksContainerResponse`  
  (`error` field + `List<TaskResponse>? data`)

- `createTasks(...)`
- `changeTaskStatus(...)`
- `deleteTask(...)`

These mutate task state and return an updated container.

### Mapping to the Domain Layer

- `TaskResponse.toDomain()` → `TaskDomain`
- `TasksContainerResponse.toDomain()` → `TasksContainerDomain`

  stores email

- `Future<void> clearEmail()`  
  removes persisted email

Used through:

- `AuthRepositoryImpl`

## Initial Setup

1. Clone the repository:

```bash
git clone <repository-url>
cd flutter_soft_wars
```

2. Install dependencies:

```bash
flutter pub get
```

3. (Optional) Regenerate Retrofit / `json_serializable` / `injectable` code if models or DI have changed:

```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

4. Make sure the backend specified in `DevelopmentEnvironment.tasksBaseUrl` is available:

```dart
String tasksBaseUrl = 'https://to-do.softwars.com.ua';
```

If a different environment is needed, create your own `AppEnvironment` implementation with the appropriate qualifier (`@prod`, `@stage`, etc.) and call:

```dart
configureDependencies(Environment.prod)
```

in `main.dart`.

## After Launch

- The first screen is `LogInPage` with a **Sign In** button.

- After tapping it:
  - an email input dialog appears
  - email is validated
  - on success it is stored in `SharedPreferences`
  - user is redirected to `HomePage` with the task list

### On `HomePage`

- In `initState`, `TasksLoadEvent` is dispatched and loads tasks for the current email.

- Available actions:
  - filter tasks (`All / Work / Personal`)
  - tap `+` to open `TaskPage` for creating a new task

### On `TaskPage`

You can:

- set title and description
- choose task type (work/personal)
- attach an image
- select due date
- mark task as urgent

Saving and deleting use `TaskBloc`;  
in case of errors, dialogs with messages are shown.
